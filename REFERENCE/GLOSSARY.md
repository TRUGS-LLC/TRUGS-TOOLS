# TRUGS Toolchain Glossary

> Alphabetical reference of all key terminology used in the TRUGS v1.0 protocol, Python toolchain, and graph storage system. 133 terms covering protocol concepts, node/edge fields, branch vocabularies, CLI commands, graph storage, and maintenance workflows.

---

**`/add-edge`** — Claude Code skill that creates edges between files. Smart mode reads a file, infers targets and relations, and proposes edges for approval. Three modes: smart (one file), directed (two files), explicit (two files + relation).

**`/maintenance`** — Claude Code skill for mid-session checkpoints. Captures decisions, creates edges, updates EPIC task statuses. Additive — maintenance creates.

**`/stop`** — Claude Code skill for end-of-day shutdown. Runs maintenance, then cleans orphaned edges, removes dead stale nodes, syncs EPIC, runs integrity check. Subtractive — stop cleans.

**AAA.md** — Eight-phase development tracker document (Vision → Maintenance) generated by the renderer. Serves as the single source of truth for task status and change logs within a folder.

**AGENT** — Node type in the Orchestration branch representing an autonomous actor that performs tasks, makes decisions, or delegates work.

**ANSWER** — Node type in the Knowledge branch representing a response or resolution to a query.

**ARCHITECTURE.md** — Dense technical reference document generated by the renderer. Intended for LLMs and technical readers as a system-level overview of a folder's structure.

**BASICBLOCK** — Node type in the LLVM branch representing a straight-line sequence of instructions with a single entry and single exit point.

**BEHAVIOR** — Node type in the Living branch representing an observable action or pattern exhibited by an instance.

**Branch** — A named vocabulary within the TRUGS protocol that defines the valid node types and edge relations for a specific domain. There are 10 branches: Python, Rust, LLVM, Web, Writer, Semantic, Orchestration, Living, Knowledge, and Executable.

**CALLS** — Edge relation indicating that one node invokes or calls another (e.g., a function calling another function).

**Bidirectional Invariant** — The TRUGS CORE rule that parent.contains[] and child.parent_id must always agree. If node A lists B in its contains[], then B.parent_id must equal A. Enforced automatically by trugs-store's add_node(parent_id=...).

**Capability** — A declaration at the graph root specifying which extensions, vocabularies, or profiles a TRUG uses. Capabilities enable opt-in features beyond the base protocol.

**CAUSES** — Edge relation in the Semantic branch indicating a causal relationship between two nodes.

**CITATION** — Node type in the Writer branch representing a formal reference to an external source within a document.

**CLASS** — Node type in the Python branch representing a Python class definition.

**CLI** — Command-line interface for the TRUGS toolchain, providing filesystem commands (`tinit`, `tadd`, `tls`, etc.) for creating and managing TRUGs.

**CLAUDE.md** — Auto-generated agent instructions file rendered from agent_instructions.trug.json by trugs-claude-render. Provides LLM agents with repository rules and conventions.

**CONCEPT** — Node type in the Semantic branch representing an abstract idea or meaning unit.

**CONTAINS** — Edge relation indicating hierarchical containment; the source node structurally contains the target node.

**`contains`** — Node field (array of strings). Lists the IDs of child nodes directly contained within this node.

**DEFINED_BY** — Edge relation linking a node to the specification or rule that defines it.

**DELEGATES_TO** — Edge relation in the Orchestration branch indicating that one agent or task hands off responsibility to another.

**DEPENDS_ON** — Edge relation indicating that the source node requires the target node to function correctly.

**Dimension** — A named numeric property attached to a node (e.g., `complexity`, `importance`). Dimensions enable quantitative analysis and filtering across the graph.

**`dimension`** — Node field (object). Holds key-value pairs of named numeric dimensions for the node.

**DOCUMENT** — Node type in the Writer branch representing a complete written document.

**Edge** — A directed connection between two nodes in a TRUG graph, defined by three fields: `from_id`, `to_id`, and `relation`.

**ENFORCED_BY** — Edge relation linking a rule or constraint to the mechanism that enforces it.

**ENTITY** — Node type in the Semantic branch representing a concrete, identifiable thing or object.

**ESCALATION** — Node type in the Orchestration branch representing an elevation of a task or issue to a higher authority or priority level.

**Executable Branch** — Branch vocabulary for representing runnable operations, including node types like OPERATION, OPERAND, RESULT, and TOOL_EXECUTION.

**EXPRESSION** — Node type in the Python branch representing a Python expression (a code fragment that evaluates to a value).

**Extension** — An optional augmentation to the TRUGS protocol that adds additional fields or semantics. The three extensions are: `typed`, `scoped`, and `ownership`.

**`from_id`** — Edge field (string). The ID of the source node where the edge originates.

**FUNCTION** — Node type in the Python branch representing a Python function definition.

**`folder-check`** — CLI command (`trugs-folder-check`) that validates folder.trug.json files for structural correctness, node type validity, edge consistency, and filesystem existence. Uses trugs-store internally.

**`folder-init`** — CLI command (`trugs-folder-init`) that generates a starter folder.trug.json by scanning the filesystem. Uses trugs-store add_node() for bidirectional invariant enforcement.

**`folder-render`** — CLI command (`trugs-folder-render`) that regenerates ARCHITECTURE.md from folder.trug.json. Uses trugs-store for loading.

**`folder-sync`** — CLI command (`trugs-folder-sync`) that updates folder.trug.json with current filesystem state. Adds nodes for new files, marks missing files stale. Uses trugs-store internally.

**`folder.trug.json`** — The single source of truth TRUG file describing a folder's contents. The renderer compiles this file into AAA.md, README.md, and ARCHITECTURE.md.

**Generator** — Toolchain component that creates new TRUG files from templates or source analysis. Produces valid `folder.trug.json` output.

**GOVERNS** — Edge relation indicating that the source node sets rules or policies that the target node must follow.

**GraphStore** — PEP 544 Protocol interface defining 22 methods for graph storage backends. Categories: Node Read (4), Node Write (5), Edge Read (4), Edge Write (3), Traversal (2), Subgraph (1), Metadata (2), Validation (1). Defined in trugs-store.

**Graph** — The top-level structure of a TRUG: a directed graph composed of nodes and edges, with optional metadata, dimensions, and capabilities.

**HITM (Human-In-The-Middle)** — Hard rule: only the human merges pull requests. Agents create PRs and return URLs but never merge. A review gate ensuring human oversight of all changes reaching the default branch.

**HAS_PROPERTY** — Edge relation in the Semantic branch linking an entity or concept to one of its properties.

**InMemoryGraphStore** — Dict-based implementation of the GraphStore Protocol. O(1) node lookup, O(degree) edge access. Used by all folder-* CLI commands via trugs-store.

**`id`** — Node field (string). A unique identifier for the node within its TRUG graph.

**IMPL** — Node type in the Rust branch representing an `impl` block that provides method implementations for a struct or trait.

**IMPORT** — Node type in the Python branch representing an import statement.

**IMPORTS** — Edge relation indicating that the source node imports or brings in the target node's definitions.

**INSTANCE** — Node type in the Living branch representing a concrete instantiation of a class, template, or pattern.

**INSTRUCTION** — Node type in the LLVM branch representing a single LLVM IR instruction within a basic block.

**INTENT** — Node type in the Semantic branch representing the purpose or goal behind an action or structure.

**JsonFilePersistence** — trugs-store class that loads .trug.json files into InMemoryGraphStore and saves them back. Replaces raw json.load/json.dump for graph I/O.

**IS_A** — Edge relation in the Semantic/Knowledge branch indicating a type or classification relationship (e.g., "Dog IS_A Animal").

**Knowledge Branch** — Branch vocabulary for representing question-answer and research structures, including QUERY, ANSWER, CITATION, and REFERENCE node types.

**LEADS_TO** — Edge relation indicating a sequential or logical progression from one node to another.

**`lifetime` Extension** — Protocol extension that adds lifetime annotations to nodes, tracking creation, expiration, or validity periods.

**Living Branch** — Branch vocabulary for representing runtime and dynamic structures, including INSTANCE and BEHAVIOR node types.

**LLVM Branch** — Branch vocabulary for representing LLVM intermediate representation, including BASICBLOCK, INSTRUCTION, and OPERAND node types.

**Metric Level** — A combined label in the format `{SI_PREFIX}_{SEMANTIC_NAME}` that indicates both the granularity and semantic role of a node (e.g., `KILO_SPEC`, `BASE_FUNCTION`, `MILLI_STATEMENT`).

**`metric_level`** — Node field (string). The metric level label assigned to this node, combining an SI prefix with a semantic name.

**MODULE** — Node type in the Python branch representing a Python module (a single `.py` file).

**Node** — A vertex in the TRUG graph representing a discrete unit of code, content, or knowledge. Defined by seven fields: `id`, `type`, `properties`, `parent_id`, `contains`, `metric_level`, and `dimension`.

**ON_FAILURE** — Edge relation in the Executable branch representing a control-flow path taken when an operation fails.

**ON_SUCCESS** — Edge relation in the Executable branch representing a control-flow path taken when an operation succeeds.

**OPERAND** — Node type in the Executable branch representing an input value or argument to an operation.

**OPERATION** — Node type in the Executable branch representing a discrete runnable action or computation.

**Orchestration Branch** — Branch vocabulary for representing multi-agent coordination, including AGENT, PRINCIPAL, TASK, ESCALATION, and PERMISSION node types.

**`ownership` Extension** — Protocol extension that adds ownership semantics to nodes, tracking who or what owns or is responsible for a node.

**PAGE** — Node type in the Web branch representing a single web page.

**PARAGRAPH** — Node type in the Writer branch representing a paragraph of text within a document or section.

**PersistenceAdapter** — trugs-store Protocol for loading/saving graph data. Implementations: JsonFilePersistence (file I/O), PostgresPersistence (SQL). Follows the Sans-IO pattern — graph logic is separate from storage.

**PORT** — Graph-native FUSE filesystem (#686). PostgreSQL-backed, shares schema.sql with trugs-store. Go implementation that makes the graph the filesystem.

**PostgresGraphStore** — SQL-backed implementation of the GraphStore Protocol. Uses psycopg3 + SQLAlchemy Core. Optional dependency (`pip install trugs-store[postgres]`). Shares schema with PORT.

**`parent_id`** — Node field (string or null). The ID of this node's parent in the containment hierarchy; null for root nodes.

**PART_OF** — Edge relation indicating compositional membership (the inverse perspective of CONTAINS at the semantic level).

**PERMISSION** — Node type in the Orchestration branch representing an authorization grant or access-control rule.

**PRINCIPAL** — Node type in the Orchestration branch representing an authenticated identity (user, service, or system) that can be granted permissions.

**PRODUCES** — Edge relation indicating that the source node generates or outputs the target node.

**`properties`** — Node field (object). An open-ended key-value map holding additional metadata specific to the node (e.g., `name`, `docstring`, `line_number`).

**Python Branch** — Branch vocabulary for representing Python source code, including MODULE, CLASS, FUNCTION, STATEMENT, EXPRESSION, and IMPORT node types.

**QUERY** — Node type in the Knowledge branch representing a question or information request.

**README.md** — Prose quickstart and motivation document generated by the renderer. Intended for all audiences as a human-friendly introduction to a folder.

**REFERENCE** — Node type in the Writer or Knowledge branch representing a pointer to related material or source.

**REFERENCES** — Edge relation indicating that the source node refers to or mentions the target node without importing or containing it.

**`relation`** — Edge field (string). The type of relationship the edge represents (e.g., CONTAINS, CALLS, IMPORTS).

**Renderer** — Toolchain component that deterministically compiles a `folder.trug.json` into AAA.md, README.md, and ARCHITECTURE.md.

**REPORTS_TO** — Edge relation in the Orchestration branch indicating a reporting or supervisory relationship.

**REQUIRES** — Edge relation indicating that the source node has a hard dependency on the target node being present or satisfied.

**RESOURCE** — Node type in the Orchestration branch representing a managed asset such as a file, service, or infrastructure component.

**RESULT** — Node type in the Executable branch representing the output or return value of an operation.

**Rust Branch** — Branch vocabulary for representing Rust source code, including MODULE, STRUCT, IMPL, FUNCTION, and STATEMENT node types.

**`scoped` Extension** — Protocol extension that adds lexical or logical scope information to nodes, enabling scope-aware analysis.

**SECTION** — Node type in the Web branch representing a logical section of a page.

**Semantic Branch** — Branch vocabulary for representing meaning and relationships, including CONCEPT, INTENT, and ENTITY node types.

**SI Prefix** — The granularity component of a metric level, borrowed from the SI prefix system: MEGA, KILO, HECTO, DEKA, BASE, DECI, CENTI, MILLI, MICRO, NANO (from coarsest to finest).

**SITE** — Node type in the Web branch representing an entire website or web application.

**`ssa` Extension** — Protocol extension that adds Static Single Assignment form annotations, enabling SSA-based analysis of data flow.

**STATEMENT** — Node type in the Python (and other) branches representing a single executable statement.

**STRUCT** — Node type in the Rust branch representing a Rust struct definition.

**SUPERSEDES** — Edge relation indicating that the source node replaces or overrides the target node.

**SYNTHESIS** — Node type in the Knowledge branch representing a combined or synthesized conclusion drawn from multiple sources.

**SYNTHESIZES_TO** — Edge relation indicating that multiple source nodes are combined to produce the target node.

**TASK** — Node type in the Orchestration branch representing a unit of work to be performed by an agent.

**`tadd`** — CLI filesystem command that adds a new node to an existing TRUG graph.

**`tcd`** — CLI filesystem command that changes the current working context to a different node or subgraph within a TRUG.

**`tdim`** — CLI filesystem command that views or modifies dimension values on nodes.

**TEMPLATES** — Edge relation indicating that the source node serves as a template or blueprint for the target node.

**TESTS** — Edge relation indicating that the source node tests or verifies the target node.

**`tfind`** — CLI filesystem command that searches for nodes matching specified criteria across TRUGs.

**`tinit`** — CLI filesystem command that initializes a new TRUG graph in the current directory, creating a `folder.trug.json` file.

**`tlink`** — CLI filesystem command that creates an edge (link) between two nodes in a TRUG graph.

**`tls`** — CLI filesystem command that lists nodes and edges in the current TRUG context.

**`tmove`** — CLI filesystem command that moves a node to a different parent within the TRUG graph.

**`to_id`** — Edge field (string). The ID of the target node where the edge points.

**TOOL_EXECUTION** — Node type in the Executable branch representing the invocation of an external tool or command.

**TRIGGERS** — Edge relation indicating that the occurrence of the source node causes the target node to activate.

**trugs-store** — Python package providing the GraphStore Protocol and storage backends (InMemoryGraphStore, PostgresGraphStore, JsonFilePersistence). Canonical library for all programmatic .trug.json access. Enforces bidirectional invariant automatically. Located at TRUGS_STORE/trugs-store/.

**TRUG** — A single JSON file (`.trug.json`) representing code, knowledge, or content as a directed graph of typed nodes and labeled edges.

**TRUGS** — Tree-Relational Unified Graph Structures. The overarching protocol and format for representing structured information as directed graphs with typed nodes, labeled edges, dimensions, and capabilities.

**`tsync`** — CLI filesystem command that synchronizes a TRUG graph with the actual filesystem, updating nodes to reflect added, removed, or changed files.

**`twatch`** — CLI filesystem command that monitors filesystem changes and automatically updates the corresponding TRUG graph in real time.

**`type`** — Node field (string). The classification of the node, drawn from the vocabulary of its branch (e.g., `FUNCTION`, `CLASS`, `AGENT`).

**`typed` Extension** — Protocol extension that adds explicit type annotations to nodes, enabling type-aware analysis and validation.

**`validate_graph()`** — trugs-store method that checks a graph against 8 TRUGS CORE invariants: bidirectional hierarchy, DAG (no cycles), dimension consistency, metric level ordering, dimension declaration, edge endpoint existence, required node fields, required edge fields. Returns List[Violation].

**Validation Rule** — One of the nine rules in the TRUGS v1.0 protocol that a TRUG must satisfy to be considered valid (e.g., unique IDs, valid parent references, correct edge targets).

**Validator** — Toolchain component that checks a TRUG file against the nine validation rules and reports errors or warnings. Distinct from trugs-store's validate_graph() — the validator also checks branch schema compliance.

**Violation** — trugs-store class representing a single CORE validation failure. Fields: node_id, rule, message, severity (error or warning).

**Web Branch** — Branch vocabulary for representing web content structures, including SITE, PAGE, and SECTION node types.

**Writer Branch** — Branch vocabulary for representing written documents, including DOCUMENT, PARAGRAPH, CITATION, and REFERENCE node types.
