## Purpose

`trugs folder-render` reads a `folder.trug.json` file and produces a deterministic `ARCHITECTURE.md` file. The generated file is the machine-voice representation of a folder — dense, structured, and never manually edited. Every fact in it comes from the TRUG or from static sources referenced by TRUG nodes.

This command implements the governance requirement that "ARCHITECTURE.md is the single container for all machine-derived knowledge about a folder" (SPEC_folder_governance.md, Layer 2).

---

## Architecture

### Design Principles

1. **Deterministic output** — Same TRUG input → byte-identical ARCHITECTURE.md. No random, environment-dependent, or time-dependent output (unless `render_date` is omitted, in which case `date.today()` is used).
2. **Zero runtime dependencies** — Only Python stdlib: `json`, `pathlib`, `datetime`, `typing`.
3. **Single function** — `render_architecture()` in `renderer.py` is the entry point. It accepts a dict, path, or JSON string.
4. **Extend, not replace** — The existing `render_architecture()` function is extended with new sections; the core rendering logic is preserved.

### Module Structure

| File | Role |
|------|------|
| `trugs_tools/renderer.py` | `render_architecture()` — the render function |
| `trugs_tools/cli.py` | `folder_render_command()` — CLI subparser |
| `trugs_tools/filesystem/folder_check.py` | `find_all_folder_trugs()` — reused for `--all` |
| `tests/test_folder_render.py` | Snapshot, unit, and CLI tests |
| `tests/fixtures/folder_render/` | Test fixture TRUGs and expected outputs |

### Rendering Pipeline

```
folder.trug.json → _load_trug() → render_architecture() → ARCHITECTURE.md
```

1. Load TRUG (dict, path, or JSON string)
2. Extract FOLDER root node and children
3. Render each section (S-01 through S-12) in order
4. Return joined markdown string

## Output Sections

### S-01: Generated Header

**Source:** Static text + `render_date` parameter  
**Format:**
```markdown
<!-- GENERATED BY TRUGS — DO NOT EDIT -->
<!-- Source: folder.trug.json | Generated: {render_date} -->

# {name} Architecture

> **For Development Context:** See [AAA.md](AAA.md)
> **For Human-Friendly Docs:** See [README.md](README.md)
```
**Sort order:** N/A  
**Empty case:** Always rendered.

### S-02: Trust Notices

**Source:** DOCUMENT nodes with `stale` and/or `verified` properties  
**Format:**

| `verified` | `stale` | Display |
|---|---|---|
| `true` | `false` | `> ✅ VERIFIED: {name} confirmed current by {verified_by} on {verified_date} (TRUG: {trug_hash})` |
| `false` | `false` | (no notice — default unreviewed state) |
| `false` | `true` | `> ⚠️ STALE: {name} {stale_reason}` + last verified + TRUG hash + "Human action required" |
| `true` | `true` | `> 🚨 CONFLICT: {name} was verified but drift detected. {stale_reason}` + last verified + TRUG hash + "Re-verify" |

**Sort order:** DOCUMENT nodes sorted by (type, id)  
**Empty case:** No trust notices section rendered if no DOCUMENT nodes have stale or verified flags set.

### S-03: Quick Reference

**Source:** FOLDER node properties (`phase`, `status`, `version`, `purpose`), TRUG top-level (`type`, node count, edge count), dimensions, capabilities  
**Format:**
```markdown
**Version:** {version}
**Purpose:** {purpose}
**Phase:** {phase}
**Status:** {status}
**Source:** `folder.trug.json`

---

## Quick Reference

**What:** {purpose}
**Type:** {type}
**Nodes:** {node_count}
**Edges:** {edge_count}
**Components:** {type}: {count}, ...
```
**Sort order:** Component types sorted alphabetically  
**Empty case:** Always rendered.

### S-04: Component Inventory

**Source:** All TRUG nodes  
**Format:**
```markdown
## Component Inventory

| Type | Name | Purpose |
|------|------|---------|
| {type} | {name} | {purpose} |
```
**Sort order:** Nodes sorted by (type, id)  
**Empty case:** Always rendered (at minimum, the FOLDER node).

### S-05: Component Hierarchy Tree

**Source:** FOLDER node + direct children  
**Format:**
```markdown
## Component Hierarchy

```
{name}/ (FOLDER, {metric_level})
  ├── {child_name} ({child_type}, {child_metric_level})
  └── {last_child_name} ({last_child_type}, {last_child_metric_level})
```
```
**Sort order:** Children sorted by (type, name) via `_get_children()`  
**Empty case:** Always rendered (at minimum, the FOLDER root line).

### S-06: Node Detail Table

**Source:** All TRUG nodes  
**Format:**
```markdown
## Node Details

| ID | Type | Name | Metric Level | Purpose |
|----|------|------|-------------|---------|
| `{id}` | {type} | {name} | {metric_level} | {purpose} |
```
**Sort order:** Nodes sorted by (type, id)  
**Empty case:** Always rendered.

### S-07: Cross-Folder Dependencies

**Source:** Edges where `to_id` contains `:` (format: `folder_name:node_id`)  
**Format:**
```markdown
## Cross-Folder Dependencies

| From | Relation | Target Folder | Target Node | Weight |
|------|----------|---------------|-------------|--------|
| {from_name} | {relation} | {target_folder} | {target_node} | {weight} |
```
**Sort order:** Edges sorted by (relation, from_id, to_id)  
**Empty case:** Section omitted entirely when no cross-folder edges exist.

### S-08: Internal Relationships

**Source:** Edges where `to_id` does NOT contain `:`  
**Format:**
```markdown
## Relationships

| From | Relation | To | Weight | Description |
|------|----------|----|--------|-------------|
| {from_name} | {relation} | {to_name} | {weight} | {description} |
```
**Sort order:** Edges sorted by (relation, from_id, to_id)  
**Empty case:** Section omitted entirely when no internal edges exist.

### S-09: Dependency Graph

**Source:** All edges (internal + cross-folder)  
**Format:**
```markdown
## Dependency Graph

```
{from_name} --[{relation}, {weight}]--> {to_name_or_cross_folder_id}
```
```
**Sort order:** Edges sorted by (relation, from_id, to_id)  
**Empty case:** Section omitted entirely when no edges exist.

### S-10: Test Summary

**Source:** TEST_SUITE node properties (`test_count`, `coverage`, `test_files`)  
**Format:**
```markdown
## Test Summary

| Test Suite | Test Count | Coverage | Test Files |
|------------|-----------|----------|------------|
| {name} | {test_count} | {coverage} | {test_files} |
```
**Sort order:** TEST_SUITE nodes sorted by (type, id)  
**Empty case:** Section omitted entirely when no TEST_SUITE nodes exist.

### S-11: File Statistics

**Source:** Nodes with `file_count`, `loc`, or `files` properties  
**Format:**
```markdown
## File Statistics

| Component | File Count | LOC |
|-----------|-----------|-----|
| {name} | {file_count} | {loc} |
```
**Sort order:** Nodes sorted by (type, id)  
**Empty case:** Section omitted entirely when no nodes have file count properties.

### S-12: Node Properties

**Source:** All node properties  
**Format:**
```markdown
## Node Properties

### {name} ({type})

- **{key}:** {value}
```
**Sort order:** Nodes sorted by (type, id). Properties sorted alphabetically by key.  
**Empty case:** Always rendered. Shows "*No properties*" for nodes without properties.

Additionally, between S-11 and S-12, Specification and Example links are rendered:

**Specifications:** From SPECIFICATION nodes. Links to `.md` files are rendered as markdown links.  
**Examples:** From EXAMPLE_SET nodes. Shows example count if available.

Both sections are omitted when no relevant nodes exist.

## CLI Interface

### Usage

```
trugs folder-render PATH            Render ARCHITECTURE.md for one folder
trugs folder-render --all           Render ARCHITECTURE.md for all folders
trugs folder-render PATH --dry-run  Print to stdout instead of writing file
trugs folder-render PATH --output FILE  Write to custom path instead of ARCHITECTURE.md
```

### Options

| Option | Description |
|--------|-------------|
| `PATH` | Path to folder containing `folder.trug.json`, or to `folder.trug.json` itself |
| `--all` | Find and render all `folder.trug.json` files (reuses `find_all_folder_trugs()`) |
| `--dry-run` | Print to stdout, do not write files |
| `--output FILE` | Write to custom path instead of `ARCHITECTURE.md` |
| `--root DIR` | Root directory for `--all` scanning (default: cwd) |
| `--render-date DATE` | Fixed date string for deterministic output (e.g., `2026-01-01`) |

### Output Behavior

- **Single folder:** Writes `ARCHITECTURE.md` in the same directory as `folder.trug.json`
- **`--all`:** Writes `ARCHITECTURE.md` in each folder that has a `folder.trug.json`. Prints summary: "Rendered N ARCHITECTURE.md files"
- **`--dry-run`:** Prints to stdout, does not write files. For `--all`, prefixes each output with `--- {folder_path} ---`
- **`--output`:** Only valid for single folder mode. Writes to specified path.

### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | TRUG file not found or invalid |
| 2 | Write error |

## Snapshot Test Strategy

### Test Fixtures

Located in `tests/fixtures/folder_render/`:

| Fixture | Purpose |
|---------|---------|
| `minimal.json` | FOLDER + 1 DOCUMENT. Verifies skeleton output. |
| `full_featured.json` | All 8 node types, cross-folder edges, file stats. Verifies all sections. |
| `stale_docs.json` | DOCUMENT nodes with stale/verified/conflict states. Verifies trust notices. |
| `cross_folder.json` | Edges with `to_id` containing `:`. Verifies cross-folder section. |
| `empty_collections.json` | No edges, no TEST_SUITE, no EXAMPLE_SET. Verifies empty section omission. |

### Comparison Rules

1. Render with fixed `render_date` (e.g., `"2026-01-15"`) for determinism
2. Assert specific section headings present/absent
3. Assert specific content strings in output
4. Assert ordering constraints (trust notices before Quick Reference, types sorted alphabetically)

### Coverage Target

≥90% coverage on renderer.py's folder-render code paths.
